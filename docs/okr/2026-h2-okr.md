# 2026 H2 OKR: MoE Recommendation Scaling Law

Date range: 2026-06-21 to 2026-12-31.

## Objective 1: Build a reproducible research mega repo

Create one versioned workspace that can hold research plans, code, experiment
metadata, literature, infrastructure notes, and artifacts without ambiguity.

Key results:

- KR1: All new experiments use the standard experiment ID, config, artifact
  manifest, and result summary format.
- KR2: Literature, model configs, dataset descriptors, and remote resources are
  tracked in `registry/` with owners and update dates.
- KR3: The repo has a documented policy for what belongs in Git, what belongs in
  Git LFS or object storage, and what must never be committed.
- KR4: A GitHub remote exists and the default branch is protected before broader
  collaboration begins.

## Objective 2: Turn the observed MoE-in-Rec scaling law into a rigorous result

Move from "we observed a scaling-law pattern" to a controlled, reproducible
claim that can survive internal review and external paper review.

Key results:

- KR1: Define the scaling variables: data size, train tokens/events, total
  parameters, activated parameters, number of experts, active experts, FLOPs,
  latency budget, and memory budget.
- KR2: Run a dense-vs-MoE experiment matrix with at least three scales per
  controlled axis and at least one held-out extrapolation point.
- KR3: Fit power-law-plus-constant curves and report residuals, confidence
  intervals, and failure cases instead of only best curves.
- KR4: Produce an internal technical report with plots, methodology, and
  reproducibility instructions.

## Objective 3: Identify publishable systems contributions

Use the scaling-law evidence to motivate infrastructure work that matters to
large-scale recommender training and serving.

Key results:

- KR1: Profile MoE recommendation bottlenecks across routing, expert imbalance,
  all-to-all communication, embedding access, attention compute, and serving
  latency.
- KR2: Build at least one prototype that improves the scaling frontier under a
  fixed training or serving budget.
- KR3: Compare against strong dense and MoE baselines with production-relevant
  metrics, not only offline loss.
- KR4: Draft a paper outline that separates modeling insight, scaling law, and
  infrastructure contribution.

## Objective 4: Prepare for GPU cloud and remote execution

Make future SSH/GPU/cloud work reproducible and auditable without leaking
secrets.

Key results:

- KR1: Maintain a redacted remote inventory in `registry/remotes.yaml`.
- KR2: Every remote run records git commit, environment, data snapshot, command,
  and artifact location.
- KR3: Add bootstrap scripts for fresh GPU machines after the first remote target
  is available.
- KR4: Define a minimum security checklist for SSH keys, cloud credentials, and
  private datasets.
